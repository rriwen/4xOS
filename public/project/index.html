ChatBI

面向企业内部的 AI 智能数据分析平台，融合面向业务人员的对话式分析助手与面向数据团队的指标配置与管理后台，打通从数据定义到业务洞察的全链路体验。

01

产品价值定位

智能分析助手 —— 面向业务人员

通过自然语言交互，让运营、产品、市场等业务角色无需编写 SQL 即可自主完成数据查询与分析，一键获取可视化图表和数据解读，实现"对话即洞察"。

指标配置后台—— 面向数据团队/分析师

提供统一的指标平台治理业务口径，通过可视化的指标定义、维度管理、数据源连接与权限管控能力，确保业务语义的准确性与一致性，从源头保障数据可信。

ChatBI的核心创新在于通过产品化架构，将松散的"自然语言-SQL"映射关系，重构为以业务指标层为中心的稳健链路（自然语言 → 标准化业务指标 → SQL）。这不仅大幅提升了查询的准确率与稳定性，更通过配置后台将业务知识沉淀为可复用的数据资产，使 AI 能力建立在可信、一致的数据基础之上。

02

项目背景：从两个"不理解"开始的挑战

业务不理解技术

内外部的业务人员有强烈的数据需求，比如用户行为埋点数据分析，但无法理解和使用 SQL 取数、分析路径长。

技术不理解业务

现有的内部 Text2SQL 技术直接通过翻译自然语言拼接 SQL 的方式有局限性，准确率仅25%，无法实际用于业务。

我的角色

作为同时负责产品规划与体验设计的项目核心，我主动打破角色与能力的边界，通过技术理解与产品化思维，探索多个创新方案，系统性推动项目从概念验证走向落地实践。

03

核心贡献：产品架构与交互设计双轨创新

产品策略层：设计"业务指标层"，重构产品逻辑

问题识别

单纯优化提示词无法解决根本的语义鸿沟。

我的解决方案

首创性提出在产品中增加"业务指标层"作为翻译中枢。

  • 产品化架构：将流程从"自然语言 → SQL"重构为"自然语言 → 业务指标（如销售额、留存率） → SQL"。
  • 价值：将模糊的业务口语，映射为团队预先定义、清洗明确的业务概念，从根本上降低歧义。
成果

这一产品架构决策成为准确率从25%飞跃至80%的核心杠杆。

对比图：旧流程 vs 新流程

交互设计层：发明"可视化解释"，解决AI信任危机

问题识别

即使SQL正确，业务用户因看不懂过程而不敢使用（"黑盒"问题）。

我的设计方案

创造了"生成式解释"交互范式。

  • 过程可视化：并非简单展示SQL代码，而是设计动态信息流，可视化呈现"你的问题 → 识别出的指标 → 用于查询的字段 → 生成的图表"。
  • 分支管理：当AI产生歧义时，提供多个理解路径让用户选择，赋予用户控制权。
价值

将AI的"思考过程"透明化，将技术可靠性转化为可感知的用户信任。此交互模式被公司内部多个AI产品复用。
